# Context Self Management

上下文管理不是“感觉乱了就重来”，而是判断当前 thread 是否还能支撑可靠行动。

## 三问

### 1. 这是同一条线，还是多棵树？

- 同一条线：围绕同一个目标连续推进，可以继续。
- 多棵树：多个主题混在一起，容易错把旧结论套到新任务。

### 2. 是否有干净断点？

- 有：可以总结、收束、交接。
- 没有：先把当前动作推进到最近可描述的状态。

### 3. 我是否能可靠复述当前任务？

如果不能用 3-5 行说清：

- 用户要什么。
- 已经完成什么。
- 当前卡点是什么。
- 下一步是什么。

就不要假装继续。

## 决策表

| 情况 | 动作 |
| --- | --- |
| 上下文长但主线清楚 | 继续，必要时简短摘要 |
| 话题漂移明显 | 先总结当前任务边界 |
| 当前任务有干净断点 | 用 quality-gate 收束 |
| 需要别人接手 | 用 cross-agent-handoff 五件套 |
| 关键信息缺失 | 问用户，不要猜 |

## 输出模板

```md
### Context Check

- 当前主线：...
- 已完成：...
- 未决：...
- 下一步：继续 / 收束 / 交接 / 需要用户确认
```

## 常见错误

| 错误 | 修正 |
| --- | --- |
| 上下文一长就交给别人 | 先判断是否有干净断点 |
| 把历史结论当当前事实 | 回到最近用户请求和最新消息 |
| 总结一大篇但没有下一步 | 总结必须服务于行动 |
| 不确定还继续执行 | 明确问用户或 push back |
