# Provider Token Usage 设计方案

## 目标

为 TheTower 建立 provider token usage 的完整采集、归一化、状态同步和展示链路。

完成后状态栏不再只能显示：

```text
Commander Zavala
完成
Token --
```

而是在 provider 能提供真实 usage 时显示：

```text
Commander Zavala
回复中
Context 18.2k / 128k
In 4.1k · Out 900
```

如果 provider 不提供 usage，仍显示 `Token --`，不伪造数据。

Cat Cafe 的数据流：

```text
Provider 原始事件
-> provider service/parser 解析成 metadata.usage
-> invocation 层统一处理 metadata.usage
-> 发 invocation_usage system_info
-> 前端消费 system_info，写入 catInvocations[catId].usage
-> 状态栏 / token 卡片展示和聚合
```

TheTower 的对应数据流应保持我们自己的 runtime status 架构：

```text
Provider 原始事件
-> AgentRunner 解析成 AgentEvent.token_usage
-> CommunicationService 统一处理 token_usage
-> AgentRuntimeStatusRegistry.setTokenUsage()
-> SSE agent.token_usage
-> AgentStatusBar / future token panel 展示
```

也就是说，沿用 Cat Cafe 的 usage 字段语义和 provider parser 经验，但不沿用 `system_info JSON` 承载方式。

## 核心原则

1. provider usage 只能由 runner/provider parser 产生。
2. 前端不估算 token。
3. `CommunicationService` 不猜 provider token，只负责归一化事件流和写入 registry。
4. usage 和 context health 分开表达。
5. 聚合用量和当前上下文填充量分开表达。
6. cumulative-only usage 不能用于 context fill 判断。
7. unknown provider 必须显示 `Token --`，不能为了 UI 好看伪造值。

## TokenUsage 字段语义

参考 Cat Cafe，TheTower 的 `AgentTokenUsage` 应扩展为：

```ts
export interface AgentTokenUsage {
  /** Total input tokens, normalized across providers. May be aggregated across turns. */
  inputTokens?: number;

  /** Output tokens generated by the provider. */
  outputTokens?: number;

  /** Provider only exposes a total count, or total is separately available. */
  totalTokens?: number;

  /** Subset of inputTokens served from cache. */
  cacheReadTokens?: number;

  /** Subset of inputTokens written into cache. Claude exposes this. */
  cacheCreationTokens?: number;

  /** Reasoning tokens where available. OpenAI Responses-style APIs may expose this. */
  reasoningTokens?: number;

  /** Provider-reported or estimated cost. */
  costUsd?: number;
  costEstimated?: boolean;

  /** End-to-end provider call duration. */
  durationMs?: number;

  /** Pure API duration where provider exposes it. */
  durationApiMs?: number;

  /** Number of model turns inside this invocation, where available. */
  numTurns?: number;

  /** Context window capacity. Exact if provider reports it; fallback if resolver supplies it. */
  contextWindowSize?: number;

  /**
   * Last API turn's total input tokens.
   * This is better than aggregated inputTokens for context fill.
   */
  lastTurnInputTokens?: number;

  /**
   * True when token stats are cumulative across the provider session and not safe
   * for per-turn context fill.
   */
  isCumulativeUsage?: boolean;

  /** Current provider session context usage, e.g. Codex token_count. */
  contextUsedTokens?: number;

  /** Reset timestamp for provider context/session quota display. */
  contextResetsAtMs?: number;

  /** The resolved budget/window used by TheTower UI. */
  budgetTokens?: number;

  /** budgetTokens - context fill or total usage, depending on display mode. */
  remainingTokens?: number;

  source: "provider" | "estimated" | "unavailable";
}
```

Cat Cafe 的 key lesson：`inputTokens` 不总是“当前上下文长度”。它可能是跨 turns 聚合值。因此 TheTower 展示 context 时不能简单用 `inputTokens / budgetTokens`。

## Usage 与 Context Health 的区别

### Provider usage

Provider usage 回答的是：

```text
这次调用用了多少输入、输出、缓存、费用？
```

适合展示：

```text
In 4.1k · Out 900 · Cache 3.2k
```

适合聚合：

```text
本轮 route total: In 12.4k · Out 2.1k
```

### Context health

Context health 回答的是：

```text
这次模型调用实际上下文填充了多少？离窗口上限还有多远？
```

适合展示：

```text
Context 18.2k / 128k · 86% left
```

推荐 context fill 解析优先级：

```text
contextUsedTokens
?? lastTurnInputTokens
?? inputTokens
?? totalTokens
```

但如果 `isCumulativeUsage === true` 且没有 `lastTurnInputTokens` / `contextUsedTokens`，不能计算 context health。

## Provider 解析策略

### Claude

Cat Cafe 做法：

- 从 Claude result / success event 读取 `usage`。
- 把 `input_tokens + cache_read_input_tokens + cache_creation_input_tokens` 归一化为 `inputTokens`。
- 读取 `output_tokens`。
- 读取 `total_cost_usd`、`duration_ms`、`duration_api_ms`、`num_turns`。
- 从 `modelUsage` / `model_usage` 中读取 `contextWindow`。

TheTower 实现建议：

```ts
export function extractClaudeUsage(event: Record<string, unknown>): AgentTokenUsage {
  const usage = event.usage as Record<string, unknown> | undefined;
  const rawInput = number(usage?.input_tokens) ?? 0;
  const cacheRead = number(usage?.cache_read_input_tokens) ?? 0;
  const cacheCreate = number(usage?.cache_creation_input_tokens) ?? 0;

  return {
    inputTokens: rawInput + cacheRead + cacheCreate,
    outputTokens: number(usage?.output_tokens),
    cacheReadTokens: cacheRead || undefined,
    cacheCreationTokens: cacheCreate || undefined,
    costUsd: number(event.total_cost_usd),
    durationMs: number(event.duration_ms),
    durationApiMs: number(event.duration_api_ms),
    numTurns: number(event.num_turns),
    contextWindowSize: extractClaudeContextWindow(event),
    source: "provider",
  };
}
```

### Codex

Cat Cafe 做法：

- 从 `turn.completed` 读取：
  - `usage.input_tokens`
  - `usage.output_tokens`
  - `usage.cached_input_tokens`
- 把 `input_tokens` 也写入 `lastTurnInputTokens`。
- 如果能拿到 Codex session token_count，再用它覆盖 context fill：
  - `contextUsedTokens`
  - `contextWindowSize`
  - `lastTurnInputTokens`
  - `inputTokens = contextUsedTokens`
- 可用 pricing table 估算 cost，但标记 `costEstimated = true`。

TheTower 当前 `CodexCliRunner` 使用 `codex exec --output-last-message`，这条路径只能拿最终文本，拿不到 `turn.completed`。因此分两步：

1. 当前版本：如果没有 structured output，保持 `source = "unavailable"`。
2. 后续版本：改成解析 Codex JSONL / structured stream，再按 Cat Cafe 逻辑抽取 usage。

建议 TheTower 的 Codex parser 目标事件：

```text
turn.completed
session token_count / context snapshot
assistant delta
tool call
```

在没有 token_count 时，`turn.completed.input_tokens` 可以作为 usage，但 context health 要标记为 approximate。

### Gemini

Cat Cafe 做法：

- 从 result success 的 `stats` 读取：
  - `total_tokens`
  - `input_tokens`
  - `output_tokens`
  - `cached_input_tokens`
  - `context_window`
- 标记 `isCumulativeUsage = true`，因为 Gemini CLI stats 可能是 session 累计。
- 如果没有 per-turn signal，不用它做 context health。

TheTower 实现建议：

```ts
usage.isCumulativeUsage = true;
```

如果后续拿到 per-turn input，应写入：

```ts
usage.lastTurnInputTokens = perTurnInput;
```

### OpenAI API / Custom Provider

如果后续接 OpenAI Responses API，可映射：

```text
usage.input_tokens -> inputTokens
usage.output_tokens -> outputTokens
usage.input_tokens_details.cached_tokens -> cacheReadTokens
usage.output_tokens_details.reasoning_tokens -> reasoningTokens
```

如果 provider 不提供 usage：

```ts
{ source: "unavailable" }
```

## TheTower 实现链路

### 1. 扩展共享类型

文件：

```text
packages/shared/src/index.ts
```

扩展 `AgentTokenUsage`，补齐 Cat Cafe 关键字段。

### 2. Runner 产生 token_usage event

`AgentEvent` 已有：

```ts
| { type: "token_usage"; usage: AgentTokenUsage }
```

每个 runner 的规则：

- 能解析 provider usage：`source = "provider"`。
- 只能估算 cost 或 context：对应字段标记 estimated，不覆盖真实 usage。
- 完全拿不到：不发 usage，或发 `{ source: "unavailable" }`。

推荐先做：

1. `ClaudeCliRunner` usage parser。
2. `CodexCliRunner` structured stream 解析前先保持 unavailable。
3. `MockRunner` 在测试模式下发 deterministic usage，用来验证链路。

### 3. CommunicationService 统一处理

当前 TheTower 已有：

```text
AgentEvent.token_usage
-> runtimeStatuses.setTokenUsage()
-> events.publish(agent.token_usage)
```

需要增强：

- 在 Agent invocation 开始时写入 budget/window snapshot。
- token usage 到来时合并 provider usage 和 budget。
- 如果 usage 包含 context fields，派生 context display 字段。

建议新增 helper：

```ts
resolveContextFill(usage: AgentTokenUsage): {
  usedTokens?: number;
  windowTokens?: number;
  source: "exact" | "approx" | "unavailable";
}
```

解析优先级：

```text
contextUsedTokens -> exact
lastTurnInputTokens -> exact if contextWindowSize exists, otherwise approx
inputTokens -> approx
totalTokens -> approx
```

如果 `isCumulativeUsage` 且没有 `contextUsedTokens` / `lastTurnInputTokens`：

```text
source = unavailable
```

### 4. Token budget resolver

新增文件建议：

```text
packages/api/src/agents/TokenBudgetResolver.ts
```

职责：

- 根据 `agent.provider` / `agent.model` 解析 context window。
- 支持 env override。
- 支持 unknown fallback。

优先级：

```text
Agent 显式配置 tokenBudget
-> provider/model fallback table
-> env override
-> undefined
```

注意：budget/window 是容量，不是实际 usage。不要把它和 provider usage 混为一谈。

### 5. AgentRuntimeStatusRegistry 合并 usage

当前 registry 已有 `setTokenUsage()`。需要增强：

- 数字字段合并策略区分“累加字段”和“最新快照字段”。
- `inputTokens`、`outputTokens`、`cacheReadTokens`、`cacheCreationTokens`、`costUsd`、`durationMs`、`durationApiMs`、`numTurns` 可累加。
- `contextWindowSize`、`lastTurnInputTokens`、`contextUsedTokens`、`contextResetsAtMs` 应保留最新值。
- `isCumulativeUsage` 保留最新 provider 标记。

参考 Cat Cafe `mergeTokenUsage()`：

```text
numeric aggregate:
  inputTokens
  outputTokens
  totalTokens
  cacheReadTokens
  cacheCreationTokens
  costUsd
  durationMs
  durationApiMs
  numTurns

latest snapshot:
  contextWindowSize
  lastTurnInputTokens
  contextUsedTokens
  contextResetsAtMs
```

TheTower 要额外保留：

```text
source
budgetTokens
remainingTokens
```

### 6. SSE

继续使用：

```text
agent.token_usage
```

事件里携带完整 `AgentRuntimeStatus` snapshot。

不要新增 `system_info` message。

### 7. 前端展示

当前状态栏显示：

```text
Token 12.4k / 80k
```

建议升级成两层：

紧凑状态栏：

```text
Context 18.2k / 128k
In 4.1k · Out 900
```

tooltip / 详情：

```text
Cache 3.2k
Reasoning 600
Cost $0.03
API 8.2s
source provider
```

如果只有 totalTokens：

```text
Total 12.4k
```

如果 cumulative-only：

```text
Token 42k cumulative
Context --
```

如果 unavailable：

```text
Token --
```

## 线程 / Route 聚合

Cat Cafe 的并发状态栏会聚合 active cats：

```text
inputTokens += usage.inputTokens
outputTokens += usage.outputTokens
if totalTokens && no inputTokens -> inputTokens += totalTokens
costUsd += usage.costUsd
```

TheTower 可以对应实现：

```text
per-agent usage: AgentStatusBar
thread active total: Thread header / InvocationPanel
route total: Invocation detail
```

聚合用途：

- 展示成本趋势。
- 判断一次 fanout/serial 的总 token 消耗。
- 不用于判断某个 Agent 是否会爆上下文。

爆上下文判断必须基于 per-agent context fill。

## 开发步骤

### Phase A：类型与 registry 合并策略

1. 扩展 `AgentTokenUsage` 字段。
2. 增强 `AgentRuntimeStatusRegistry.setTokenUsage()`。
3. 增加 `mergeAgentTokenUsage()`。
4. 增加 context fill resolver。
5. 增加 registry 单元测试。

### Phase B：Mock usage 验证链路

1. `MockRunner` 发 deterministic `token_usage`。
2. 确认 `agent.token_usage` SSE 有完整 snapshot。
3. 前端状态栏从 `Token --` 变成真实数字。
4. 增加 CommunicationService 测试。

### Phase C：Claude usage parser

1. 新增 `ClaudeUsageParser`。
2. 从 Claude result event 抽取 input/output/cache/cost/duration/window。
3. `ClaudeCliRunner` yield `token_usage`。
4. 增加 parser 测试和 runner 测试。

### Phase D：Codex structured usage

1. 调研当前 Codex CLI 是否可输出 JSONL / structured event。
2. 如果支持，解析 `turn.completed`。
3. 如果支持 token_count，写入 `contextUsedTokens` / `contextWindowSize`。
4. 如果不支持，保持 `source = "unavailable"`。

### Phase E：前端展示升级

1. `statusFormat.ts` 增加 context / input / output 格式化。
2. `AgentStatusBar` 展示 Context + In/Out。
3. 增加 tooltip 展示 cache/cost/duration/source。
4. thread / invocation panel 增加 route usage total。

### Phase F：Context health 与预算

1. 新增 `TokenBudgetResolver`。
2. invocation 开始时写入 context window/budget。
3. usage 到来时计算 remaining。
4. cumulative-only usage 不产生 context health。

## 验收标准

- Provider usage 可用时，状态栏不再显示 `Token --`。
- Claude usage 中 cache tokens 被计入 inputTokens，并单独展示 cacheReadTokens。
- Codex 无 structured usage 时仍显示 `Token --`，不伪造数据。
- Codex 有 token_count 时展示 `Context used / window`。
- Gemini cumulative-only usage 不用于 context fill。
- 前端能区分：
  - `In/Out usage`
  - `Context fill`
  - `Total-only usage`
  - `Unavailable`
- route/thread 聚合不影响 per-agent context 判断。
- 所有 usage SSE 事件仍不写入 MessageStore，不进入 Agent 上下文。

## 不做事项

- 不把 provider usage 写成 thread message。
- 不用前端估算 token。
- 不把 cumulative session usage 当成本次 context fill。
- 不把 cost estimated 当成 provider-reported exact cost。
- 不在没有 Codex structured output 时伪造 Codex token usage。
